# 0단계. 시작 전에 고정할 것

이 단계는 코드를 거의 안 짜더라도 먼저 해야 해.

이걸 안 하면 나중에 계속 뜯어고치게 돼.

## 목표

AI 파트에서 흔들리면 안 되는 “기준 계약”을 먼저 문서화한다.

## 지금 고정해야 하는 것

1. 공개 API 목록
2. NestJS ↔ FastAPI 내부 API 목록
3. DB 영구 저장 필드
4. Redis 임시 저장 필드
5. 삭제 정책
6. 면접 운영 규칙
7. 점수/리포트 구조
8. 에러/실패 상태 정의

문서상 이미 정해진 핵심 기준은 이거야.

- 문서 충분도: 충분 / 보통 / 부족
- 질문 구성: 총 10문항, 꼬리질문 최대 2개
- STT 실패 2회 후 텍스트 전환
- Vision은 감정 분석 아님, 보조 비언어 지표만 사용
- `raw transcript`, `raw vision metrics`, `hidden score`, `세션 중간 상태`는 10분 내 삭제
- `interview_sessions`, `interview_turns`, `answer_full_text`, `feedback_text`, 최종 리포트는 저장

## 산출물

- `docs/ai-contract.md`
- `docs/interview-rules.md`
- `docs/storage-policy.md`

## 완료 기준

- 이 문서만 보면 “무엇을 저장하고, 무엇을 버리고, 어떤 API가 필요한지” 설명 가능
- 이후 코드는 이 문서를 그대로 옮기는 작업이 됨

## Codex 요청 문장

```
현재 문서 기준을 바탕으로 AI 파트 구현용 계약 문서를 만들어줘.
반드시 아래 3개 파일 형태로 정리해줘.
1) docs/ai-contract.md
2) docs/interview-rules.md
3) docs/storage-policy.md

포함 내용:
- NestJS 공개 API 목록
- FastAPI 내부 API 목록
- 영구 저장/임시 저장/저장 안 함 데이터 구분
- 면접 규칙(문서 충분도, 질문 수, 꼬리질문, STT fallback, Vision invalidation, 세션 종료 규칙)
- 에러 상태 정의
- 각 API 요청/응답 예시 JSON
```

# 1단계. AI 폴더 뼈대 생성

지금 가장 먼저 해야 할 실제 구현은 **FastAPI 서버 실체 만들기**야.

Codex 결과상 현재 `ai` 폴더에는 실제 앱 코드가 없어.

## 목표

FastAPI 서버가 실행되고, 최소한 health check와 내부 라우트 뼈대가 동작하게 만든다.

## 만들 것

예상 구조는 이렇게 가는 게 좋아.

```
ai/
  app/
    main.py
    core/
      config.py
      logging.py
    api/
      routes/
        health.py
        jobs.py
        cover_letter.py
        interview.py
    schemas/
      jobs.py
      cover_letter.py
      interview.py
      common.py
    services/
      jobs/
      cover_letter/
      interview/
      stt/
      vision/
      tts/
    ports/
    adapters/
    utils/
  tests/
  requirements.txt
  .env.example
```

## 반드시 먼저 만들 라우트

- `GET /health`
- `POST /internal/jobs/analyze`
- `POST /internal/cover-letter/feedback`
- `POST /internal/interview/start`
- `POST /internal/interview/answer`
- `POST /internal/interview/finish`

## 산출물

- FastAPI 앱 실행 가능
- Swagger/OpenAPI 문서 자동 생성 가능
- 더미 응답으로라도 내부 API 5개가 호출 가능

## 완료 기준

- Postman 또는 curl로 FastAPI 라우트가 정상 응답
- 아직 LLM 없어도 괜찮음
- JSON contract만 맞으면 됨

## Codex 요청 문장

```
ai 폴더에 FastAPI 앱을 처음부터 생성해줘.
목표는 실제 AI 기능이 아니라 내부 API 뼈대를 만드는 것이다.

반드시 아래 조건을 지켜줘.
- app/main.py 엔트리포인트 생성
- /health 라우트 생성
- /internal/jobs/analyze
- /internal/cover-letter/feedback
- /internal/interview/start
- /internal/interview/answer
- /internal/interview/finish
- pydantic schema 분리
- service 레이어 분리
- 현재는 더미 응답으로 구현해도 됨
- 이후 LLM/STT/Vision을 붙이기 쉽게 폴더 구조를 나눠줘
```

# 2단계. NestJS 공개 API와 AI 내부 호출 연결

문서상 NestJS는 공개 게이트웨이 역할이고, FastAPI는 내부 AI 서버야.

그러니까 프론트는 무조건 NestJS만 때려야 해.

## 목표

NestJS 공개 API를 만들고, 내부적으로 FastAPI를 호출하게 만든다.

## 만들 것

공개 API:

- `POST /jobs/analyze`
- `POST /ai/cover-letter/feedback`
- `POST /ai/interview/sessions/start`
- `POST /ai/interview/sessions/:sessionId/answers`
- `POST /ai/interview/sessions/:sessionId/finish`
- `GET /ai/interview/sessions`
- `GET /ai/interview/sessions/:sessionId`
- `GET /ai/interview/sessions/:sessionId/turns`

## 구현 포인트

- Controller
- DTO
- Service
- Internal AI Client
- timeout / retry / error mapping
- 내부 헤더(shared secret 등)

## 산출물

- 프론트는 NestJS만 호출
- NestJS는 FastAPI로 내부 요청 전달
- FastAPI 응답을 정리해 프론트에 반환

## 완료 기준

- 로컬에서 `frontend -> NestJS -> FastAPI` 흐름이 실제로 한 번 연결됨
- FastAPI가 더미 응답이어도 됨

## Codex 요청 문장

```
NestJS에 AI 공개 API를 추가해줘.
현재 역할은 공개 API 게이트웨이이고, FastAPI를 내부 호출해야 한다.

반드시 아래를 구현해줘.
- JobsController / JobsService / DTO
- CoverLetterController / Service / DTO
- InterviewController / Service / DTO
- AiClientService를 통해 FastAPI 내부 호출
- baseURL, secret header, timeout, 에러 매핑 분리
- AppModule에 실제 등록
- Swagger 문서 노출

공개 API는 아래를 만들어줘.
- POST /jobs/analyze
- POST /ai/cover-letter/feedback
- POST /ai/interview/sessions/start
- POST /ai/interview/sessions/:sessionId/answers
- POST /ai/interview/sessions/:sessionId/finish
- GET /ai/interview/sessions
- GET /ai/interview/sessions/:sessionId
- GET /ai/interview/sessions/:sessionId/turns
```

# 4단계. Redis 임시 상태 설계

문서상 임시 상태는 분리되어야 하고, 10분 내 삭제돼야 해.

Codex도 Redis/TTL 저장이 아직 없다고 했어.

## 목표

세션 중간 상태와 raw 값을 영구 DB와 분리한다.

## Redis에 넣을 것

- 세션 중간 상태
- 현재 질문 인덱스
- 이전 질문 목록
- follow-up 횟수
- raw transcript
- raw vision metrics
- hidden score
- 실패 플래그
- cleanup deadline

## 추천 키 구조

- `interview:session:{sessionId}:state`
- `interview:session:{sessionId}:transcript:{turnId}`
- `interview:session:{sessionId}:vision:{turnId}`
- `interview:session:{sessionId}:hidden-score:{turnId}`
- `interview:session:{sessionId}:cleanup`

## TTL 원칙

- 세션 진행 중: 세션 활동마다 TTL 연장 가능
- 세션 종료 후: cleanup 예약 후 10분 TTL
- 실패 세션: 바로 cleanup 예약

## 산출물

- Redis client 연결
- set/get/delete helper
- TTL 정책 구현

## 완료 기준

- 세션 시작 시 state 저장
- 세션 종료 시 cleanup key 기록
- 10분 내 지워질 구조 확보

## Codex 요청 문장

```
NestJS 또는 FastAPI에서 사용할 Redis 임시 상태 저장 구조를 구현해줘.
아래 요구사항을 만족해야 해.
- interview session state 저장
- raw transcript 저장
- raw vision metrics 저장
- hidden score 저장
- cleanup deadline 저장
- key naming rule 통일
- TTL 설정 가능
- 세션 종료 후 10분 삭제 정책을 반영할 수 있도록 helper 작성

추가로 docs/redis-key-design.md 문서도 함께 만들어줘.
```

# 5단계. 파일 저장 추상화

현재는 로컬 업로드 폴더 저장이라고 봐야 해.

지금 비용 아끼는 단계에서는 로컬 저장으로 가도 되지만, **인터페이스는 S3 교체 가능 구조**여야 해.

## 목표

로컬 저장으로 개발하되, 나중에 S3로 쉽게 바꿀 수 있도록 한다.

## 꼭 분리해야 할 것

- `StoragePort`
- `LocalStorageAdapter`
- 나중에 `S3StorageAdapter`

## 현재 저장 대상 구분

### durable

- user_documents
- dataroom_items
- 최종 리포트 파일이 생기면 그 파일

### temp

- 면접 업로드 직후 처리용 파일
- STT/Vision 전처리용 임시 파일

## 저장 금지/비영구 원칙

- raw video 장기 저장 금지
- raw frame image 장기 저장 금지

## 산출물

- 저장 서비스 추상화
- 로컬 파일 저장 구현
- protected download 구조 유지

## 완료 기준

- 저장 경로가 하드코딩이 아니라 adapter로 분리됨
- 나중에 S3 교체 시 controller/service 거의 안 건드려도 됨

## Codex 요청 문장

```
파일 저장 구조를 추상화해줘.
현재는 로컬 디스크 저장으로 개발하되, 나중에 S3로 교체하기 쉽게 만들어야 한다.

반드시 아래를 구현해줘.
- StoragePort 인터페이스
- LocalStorageAdapter
- durable/temp 구분 저장
- protected storage key 반환
- 다운로드는 인증 후 서버 경유 방식 유지
- raw video, raw frame image는 장기 저장하지 않도록 구조 설계

그리고 storage 구조 설명 문서도 docs/storage-architecture.md 로 만들어줘.
```

# 6단계. 공고 분석 AI 구현

이제부터 진짜 AI 기능 구현이 시작돼.

자소서/면접 전에 먼저 공고 분석부터 가는 이유는, 이게 다른 기능의 기준 데이터가 되기 때문이야. 문서도 `job_analysis_requests`를 자소서/면접 재사용 기준으로 본다.

## 목표

URL 또는 수동 입력으로 JD 기준 정보를 만든다.

## 구현할 것

- 입력: `jobUrl | manualCompanyName | manualJobTitle | manualJdText`
- 출력:
    - company_name
    - job_title
    - jd_text
    - extracted_skills
    - extracted_keywords
    - status

## 실패 처리

- 링크 분석 실패 시 수동 입력 허용

## 산출물

- `/internal/jobs/analyze`
- `/jobs/analyze`
- DB 저장
- 재조회 가능

## 완료 기준

- URL 입력 실패해도 수동 입력 fallback 작동
- 이후 자소서/면접에서 이 데이터를 참조 가능

## Codex 요청 문장

```
공고 분석 기능을 구현해줘.
요구사항:
- job URL 분석 또는 수동 입력 둘 다 지원
- 결과는 company_name, job_title, jd_text, keywords로 정리
- 실패 시 수동 입력 fallback 가능
- FastAPI 내부 API와 NestJS 공개 API 둘 다 연결
- 결과를 job_analysis_requests 테이블에 저장
- 이후 자소서 피드백과 면접 세션에서 재사용할 수 있게 설계
```

# 7단계. 자소서 피드백 AI 구현

자소서 AI가 면접 AI보다 먼저인 이유는, 상태 관리가 덜 복잡해서 AI 파이프라인을 먼저 검증하기 좋아서야.

## 목표

사용자 문서 + JD 기준으로 자소서 피드백 리포트를 만든다.

## 입력

- cover letter file or text
- optional resume
- optional portfolio
- job_analysis_request_id

## 출력

- total_score
- summary
- strengths 3
- weaknesses 3
- next actions 3

이 구조는 문서와 잘 맞아.

## 구현 포인트

- 파일 파싱
- 텍스트 정규화
- 프롬프트 입력 스키마 고정
- 구조화된 JSON 응답 받기
- DB 저장
- 리포트 재조회

## 산출물

- `/internal/cover-letter/feedback`
- `/ai/cover-letter/feedback`
- `/ai/cover-letter/reports`
- `/ai/cover-letter/reports/:reportId`

## 완료 기준

- 결과가 DB에 저장됨
- 프론트에서 재조회 가능
- 문서 재사용 가능

## Codex 요청 문장

```
자소서 피드백 기능을 구현해줘.
반드시 아래를 포함해줘.
- 파일 업로드 또는 텍스트 입력 지원
- JD 기준 평가
- summary / strengths / weaknesses / next actions 구조화 반환
- Pydantic/Zod에 맞는 JSON schema 사용
- cover_letter_reports 저장
- NestJS에서 조회 API도 구현
- 프론트에서 결과 재조회 가능하게 데이터 형태를 정리
```

# 8단계. 면접 세션 시작 로직 구현

이제 면접 AI로 들어간다.

여기서 가장 먼저 해야 할 건 “질문 생성”이 아니라 **세션 시작 규칙**이야.

## 목표

면접 시작 시 문서 충분도와 세션 메타를 먼저 만든다.

## 구현할 것

- 입력: 회사명, 직무명, JD, 선택 문서 IDs
- 문서 충분도 판단:
    - 충분: JD + 사용자 문서 1개 이상
    - 보통: JD만 있음
    - 부족: JD도 없고 문서 빈약
- 첫 질문은 반드시 1분 자기소개
- `interview_sessions` 생성
- Redis state 초기화

## 산출물

- `POST /internal/interview/start`
- `POST /ai/interview/sessions/start`

## 완료 기준

- 세션 생성 시 DB row 생김

# 10단계. 답변 평가 엔진 구현

이 단계가 면접 AI의 핵심이야.

## 목표

답변을 보고 “충분 답변인지 / 꼬리질문으로 갈지 / 다음 질문으로 갈지” 결정한다.

## 문서 기준

충분성 기준:

- 질문 적합성 15/20 이상
- 구체성 15/20 이상
- 직무 적합성 12/15 이상
- 전체 내용 점수 70/85 이상
- 추가 조건 4개 중 3개 이상 만족

불충분 기준:

- 질문 의도에 직접 답하지 않음
- 본인 역할이 모호함
- 결과/근거가 없음
- 직무 연결이 약함
- 너무 추상적임

꼬리질문 우선순위:

1. 역할/기여도
2. 성과/근거
3. 직무 연결
4. 전문성 디테일
5. 협업/태도 보완

## 구현 포인트

- evaluator input schema
- structured output
- score breakdown
- is_sufficient boolean
- follow_up_reason
- next_action enum

## 산출물

- 답변 평가 서비스
- 꼬리질문 생성 서비스

## 완료 기준

- 답변 1개 넣으면 점수와 follow-up 필요 여부가 나옴
- follow-up은 반드시 부족점과 연결됨

## Codex 요청 문장

```
면접 답변 평가 엔진을 구현해줘.
반드시 아래를 포함해줘.
- score breakdown: 질문 적합성, 구체성, 근거/성과, 직무 적합성, 논리성/구조, 진정성/태도
- isSufficient 판단
- 부족한 이유 목록
- follow-up 필요 여부
- follow-up 우선순위는 역할/기여도 > 성과/근거 > 직무 연결 > 전문성 디테일 > 협업/태도
- LLM 결과는 구조화된 JSON으로 받고, 서버에서 검증해줘
```

# 11단계. STT fallback 구현

문서 기준에 매우 중요해.

## 목표

영상 답변 업로드 → STT 시도 → 실패 시 재업로드 → 2회 실패 시 텍스트 전환 흐름 구현

## 규칙

재업로드 조건:

- 전사 결과 1문장 이하 + 핵심 명사 부족
- 3초 이하 영상
- 음성 없음 또는 잡음 심함

2회 실패 시:

- 텍스트 답변 유도
- 비언어 점수 0점
- 내용 평가만 반영
- 텍스트도 `answer_full_text`로 저장

## 산출물

- STT service
- stt retry counter
- text fallback mode

## 완료 기준

- 영상 업로드 실패 흐름이 정상 처리됨
- 사용자가 막히지 않고 텍스트로 계속 진행 가능

## Codex 요청 문장

```
면접 답변 STT fallback 흐름을 구현해줘.
조건:
- 같은 턴에서 최대 2회 재업로드 허용
- 실패 조건: 1문장 이하 + 핵심 명사 부족, 3초 이하 영상, 음성 없음, 심한 잡음
- 2회 실패 시 텍스트 답변 유도
- 텍스트 전환 시 비언어 점수는 0점 처리
- 텍스트도 answer_full_text로 저장
- retry count는 session state 또는 turn state에 저장
```

- 가능하면 간단한 정면 응시 proxy

규칙:
- 얼굴 유지율 너무 낮으면 비언어 0점
- 다중 얼굴이면 해당 턴 Vision 무효

## 1. FastAPI부터 만든다

지금 AI 파트가 “아무것도 없음” 상태라면,

가장 먼저 만들 건 프롬프트가 아니라 **FastAPI 서버 실체**야.

## 2. 저장 구조부터 만든다

AI는 출력이 좋아 보여도 저장이 안 되면 서비스가 아니다.

`interview_sessions`, `interview_turns`, `cover_letter_reports`, `job_analysis_requests`가 먼저다.

## 3. Redis는 나중이 아니라 중간에 바로 넣는다

이 프로젝트는 임시 상태가 핵심이라 Redis를 빼면 나중에 다시 뜯어야 해.

## 4. Vision은 최소 기능만 먼저

주력은 텍스트 평가다. Vision은 보조다.

## 5. AWS는 마지막

지금은 AWS가 아니라 “로컬에서 문서 기준 그대로 완성”이 우선이다.

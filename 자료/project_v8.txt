
[ 온 세상이 취업 - project.txt ]

1. 프로젝트명
- 온 세상이 취업
- World_JobSearch

2. 프로젝트 한 줄 소개
- 자기소개서 피드백, AI 면접 연습, 게시판, 자료실을 하나의 플랫폼으로 묶은 로그인 기반 취업 준비 웹 서비스

3. 프로젝트 목적
- 취업 준비생이 자기소개서와 면접을 반복적으로 연습하고 피드백을 다시 확인할 수 있도록 돕는 실서비스형 플랫폼을 만드는 것이 목적이다.
- 단순 일회성 AI 호출이 아니라, 문서 업로드 → 공고 분석 → 자소서 피드백 → 면접 세션 → 결과 재조회까지 이어지는 사용자 여정을 만드는 것을 목표로 한다.

4. 해결하려는 문제
- 자기소개서와 면접 피드백은 비용이 크거나 접근성이 낮다.
- 사용자는 같은 문서를 반복해서 수정하고, 같은 회사/직무 기준으로 여러 번 비교하고 싶어한다.
- 면접은 텍스트만으로는 전달력까지 설명하기 어렵다.
- 그래서 이 프로젝트는 문서 기반 피드백과 면접 세션 이력을 저장하고 다시 볼 수 있는 구조를 가진다.

5. 핵심 서비스 범위
- 자체 로그인 / 소셜 로그인
- 게시판
- 자료실
- 사용자 개인 문서 업로드
- 공고 링크 분석
- 자소서 피드백 생성 및 리포트 저장
- 면접 세션 시작 / 진행 / 종료
- 면접 질문/답변/피드백 이력 조회

6. 이번 기준에서 가장 중요한 설계 원칙
- 면접 세션 시작 전 문서 충분도를 충분 / 보통 / 부족으로 판단한다.
- 면접 답변 품질 판단의 중심은 텍스트(answer_full_text)다.
- answer_full_text는 STT 결과를 바탕으로 생성하되, 사용자가 실제 말한 흐름을 최대한 보존하는 사용자용 저장 텍스트다.
- 말버릇, 반복어, 장황함은 평가 근거가 될 수 있으므로 제거하지 않는다.
- 허용되는 보정은 띄어쓰기, 문장부호, 명백한 STT 깨짐 수정 정도로 제한한다.
- Vision은 주 기능이 아니라 비언어 신호 보조 평가 모듈이다.
- 감정 분석은 하지 않는다.
- 면접 영상 원본과 raw frame image는 장기 저장하지 않는다.
- raw transcript, raw vision metrics, hidden score, 세션 중간 상태, 실패 세션의 임시 분석 데이터는 세션 종료 후 10분 내 삭제한다.
- 대신 question_text, answer_video_title, answer_full_text, feedback_text, nonverbal_summary_text, 최종 리포트는 저장한다.
- 여기서 answer_full_text는 사용자에게 다시 보여줄 답변 텍스트이며, raw transcript 원본 자체를 뜻하지 않는다.
- 규칙은 평가 기준을 고정하기 위한 기준이고, 최종 답변 충분성 판단은 LLM이 수행한다.

7. 사용자 관점 주요 기능

7-1. 인증
- 이메일/비밀번호 기반 자체 회원가입
- JWT 기반 로그인
- 카카오 / 네이버 OAuth 확장 가능
- 로그인 후 게시판, 자료실, 문서, AI 기능 접근 가능

7-2. 게시판
- 게시글 목록 조회
- 최신순 / 좋아요순 / 조회수순 정렬
- 게시글 생성 / 수정 / 삭제
- 댓글 / 대댓글 작성
- 좋아요 추가 / 취소

7-3. 자료실
- 운영자가 올린 자료 목록 조회
- 인증 사용자 다운로드
- MASTER 권한 사용자의 업로드 / 수정 / 삭제

7-4. 사용자 문서
- 자소서, 이력서, 포트폴리오, JD 문서 업로드
- 목록 조회, 다운로드, 삭제
- 자소서 피드백 / 면접 시작 시 기존 문서를 다시 선택해서 사용 가능

7-5. 공고 분석
- 공고 링크를 입력하면 회사명, 직무명, JD를 추출
- 링크 분석 실패 시 수동 입력 가능
- 이후 자소서 피드백과 면접 세션의 기준 정보로 재사용 가능

7-6. 자소서 피드백
- 파일 또는 텍스트를 입력
- JD + 사용자 문서 기준으로 평가
- 종합 점수, 요약, 강점 3개, 보완점 3개, 수정 방향 3개 반환
- 결과는 리포트로 저장되어 다시 조회 가능

7-7. 면접 AI
- 세션 시작 시 회사명, 직무명, JD, 자소서/이력서/포트폴리오 문서를 기준으로 문서 충분도를 먼저 판단한다.
- 충분: JD 포함 + 사용자 문서 1개 이상
- 보통: JD만 있음 → 면접은 JD + 공통 질문으로 진행, 자소서 기능은 비활성화
- 부족: JD도 없고 사용자 문서도 빈약 → AI 기능 비활성화
- 첫 질문은 반드시 1분 자기소개다.
- 기본 기준은 총 10문항이며, 각 질문당 꼬리질문은 최대 2개까지만 허용한다.
- 질문 구성은 1분 자기소개 1 / 우리 회사 지원동기 1 / 우리 JD 적합성 2 / 프로젝트 심층 2 / 다른 프로젝트 2 / 협업·문제해결 1 / 마무리 1을 기준으로 한다.
- 시스템은 STT를 통해 raw transcript를 만들고, 이를 바탕으로 사용자에게 다시 보여줄 answer_full_text를 생성한다.
- answer_full_text는 사용자가 실제 말한 흐름을 최대한 보존하며, 말버릇·반복어·장황함은 제거하지 않는다.
- 답변이 불충분하면 본인 역할/기여도 → 성과/근거 → 직무 연결 → 전문성 디테일 → 협업/태도 순으로 꼬리질문을 생성한다.
- STT 실패 시 같은 턴에서 최대 2회 재업로드를 허용하고, 이후에는 텍스트 답변으로 전환한다.
- 텍스트 답변 전환 시 비언어 평가는 0점 처리하고 내용 평가만으로 재산정한다.
- 최종적으로 질문 목록, 답변 전문, 피드백, 최종 코칭 결과를 저장한다.
- 영상 원본 자체는 장기 저장하지 않는다.

8. 면접 AI 상세 설계

8-1. 문서 충분도 원칙
- 충분: JD 포함 + 사용자 문서 1개 이상
- 보통: JD만 있음 → 면접은 JD와 공통 질문으로 진행하고 자소서 기능은 비활성화한다.
- 부족: JD도 없고 사용자 문서도 빈약 → AI 기능을 비활성화한다.

8-2. 답변 충분성 및 꼬리질문 원칙
- 질문 적합성 20점 중 15점 이상
- 구체성 20점 중 15점 이상
- 직무 적합성 15점 중 12점 이상
- 전체 내용 점수 85점 중 70점 이상
- 그리고 질문 의도에 직접 답함 / 본인 역할이 드러남 / 근거나 결과가 있음 / 직무와 연결됨 중 3개 이상 만족해야 충분 답변으로 본다.
- 아래 중 하나라도 강하게 걸리면 꼬리질문으로 보완한다.
  - 질문 의도에 직접 답하지 않음
  - 본인 역할이 모호함
  - 결과/근거가 없음
  - 직무 연결이 약함
  - 너무 추상적임
- 질문 유형별 중점은 지원동기(직무 적합성·진정성), 프로젝트 심층(구체성·근거/성과·역할), 협업/문제해결(논리성·태도) 순으로 본다.
- 최종 충분성 판단은 고정 룰을 참고하되 LLM이 수행한다.

8-3. 질문 구성 및 운영 제한
- 기본 기준은 총 10문항이다.
- 각 질문당 꼬리질문은 최대 2개까지 가능하다.
- 질문 구성은 1분 자기소개 1 / 우리 회사 지원동기 1 / 우리 JD 적합성 2 / 프로젝트 심층 2 / 다른 프로젝트 2 / 협업·문제해결 1 / 마무리 1이다.
- 같은 세션에서 동일 의미 질문 반복은 금지한다.
- 같은 토픽이라도 표현만 바꾼 질문을 남발하지 않는다.
- follow-up은 반드시 직전 답변의 부족점과 연결한다.
- 질문은 1~2문장, 120자 이내를 권장한다.

8-4. STT / Vision / 세션 종료 처리
- STT 실패 시 같은 턴에서 최대 2회까지 재업로드를 허용한다.
- 전사 결과가 1문장 이하 + 핵심 명사 부족, 너무 짧은 영상(예: 3초 이하), 음성 없음 또는 심한 잡음이면 재업로드를 유도한다.
- 2회 실패 시 텍스트 답변으로 전환한다.
- 텍스트 답변 전환 시 비언어 평가는 0점 처리하고 일반 평가는 내용 평가 기준으로만 재산정한다.
- Vision은 감정 추정을 하지 않는다.
- 얼굴 유지율이 너무 낮으면 해당 세션 비언어 점수는 0점 처리한다.
- 다중 얼굴이 검출되면 해당 턴 Vision은 무효 처리한다.
- 저조도/가림이 심하면 Vision 결과는 참고만 하고 점수 반영을 약화한다.
- 답변은 최소 10초 이상 권장하고, 3초 이하는 무효다.
- 30초 미만이면 “답변이 짧을 수 있습니다” 수준의 약한 경고를 준다.
- 2분 이상이면 경고 또는 요약 유도를 적용하고, 시스템 업로드 제한을 초과하면 재제출을 요청한다.
- 세션이 중간 종료되면 실제 진행 문항 수가 5문항 이상일 때만 리포트를 생성한다.
- 5문항 미만은 평가하지 않고 임시 데이터까지 모두 삭제한다.
- 실패 상태는 첫 질문 생성 실패, 답변 업로드 실패, STT 2회 실패 후 텍스트도 미입력, 시스템 오류로 다음 질문 생성 불가를 포함한다.

8-5. 저장 정책
- 영구 저장:
  - interview_sessions
  - interview_turns
  - question_text
  - answer_video_title
  - answer_full_text
  - feedback_text
  - nonverbal_summary_text
  - 최종 리포트
- answer_full_text는 사용자에게 다시 보여줄 답변 텍스트이며, 사용자가 실제 말한 흐름을 최대한 보존한다.
- 임시 저장 후 세션 종료 10분 내 삭제:
  - raw transcript
  - raw vision metrics
  - hidden score
  - 세션 중간 상태
  - 실패 세션의 임시 분석 데이터
- raw transcript는 원본 전사 데이터이며 장기 저장하지 않는다.
- 장기 저장하지 않음:
  - raw video
  - raw frame image

8-6. 최종 점수, 리포트, 결과 노출
- 최종 점수는 100점 만점 정수로 표시한다.
- 내용 평가는 85점, 비언어 보조 평가는 15점으로 계산한다.
- 내용 평가는 질문 적합성 / 구체성 / 근거·성과 / 직무 적합성 / 논리성·구조 / 진정성·태도를 사용한다.
- 비언어 평가는 얼굴 유지 안정성 / 정면 응시 프록시 / 움직임·가림·저조도 등 장해 요소를 사용한다.
- 등급 기준은 90~100 매우 우수 🌟 / 80~89 우수 ✅ / 70~79 보통 🙂 / 60~69 보완 필요 ⚠️ / 0~59 집중 보완 필요 🔧 이다.
- 리포트는 총점 / 전체 평가 요약 / 강점 3개 / 보완점 3개 / 연습 방향 3개 / 질문 목록-답변 / 턴별 피드백으로 구성한다.

9. 시스템 아키텍처

9-1. 전체 구조
- Frontend: Next.js
- Backend: NestJS
- AI Server: FastAPI
- DB: PostgreSQL
- 임시 저장소: Redis 또는 TTL 저장소
- 파일 저장소: 비공개 스토리지(S3 호환 또는 서버 스토리지)

9-2. 역할 분리
- Next.js:
  - 로그인/회원가입 화면
  - 게시판/자료실/문서/리포트/면접 세션 UI
- NestJS:
  - 인증/인가
  - 게시판/자료실/문서 API
  - AI 요청 오케스트레이션
  - FastAPI 내부 호출
- FastAPI:
  - 공고 분석
  - 자소서 피드백 생성
  - 면접 세션 진행
  - STT/Vision/TTS 연동
  - 최종 코칭 결과 생성

9-3. 비공개 원칙
- FastAPI는 외부에 직접 공개하지 않는다.
- 반드시 NestJS를 통해서만 호출한다.

11. 데이터 구조 설명

11-1. users
- 모든 사용자 계정의 기본 테이블
- 자체 로그인 + 소셜 로그인 확장 고려

11-2. posts / post_likes / comments
- 커뮤니티 기능 담당

11-3. dataroom_items
- 운영자가 관리하는 자료실 파일
- 공용 자료 영역

11-4. user_documents
- 사용자가 올린 개인 문서 저장
- 자료실과 성격이 다르므로 분리
- 자소서, 이력서, 포트폴리오, JD 문서 관리

11-5. job_analysis_requests
- 공고 분석 입력과 정리된 JD 저장
- 이후 자소서/면접에서 재사용 가능

11-6. cover_letter_reports
- 자소서 피드백의 최종 사용자 결과 저장
- 원문 전체 추론 로그가 아니라, 사용자에게 다시 보여줄 결과 중심 저장

11-7. interview_sessions
- 면접 세션의 메타 정보와 최종 결과 저장
- 회사명, 직무명, 상태, 총 질문 수, 최종 점수, 최종 결과 요약 포함

11-8. interview_turns
- 세션별 질문과 사용자 결과를 저장
- 질문 텍스트
- 답변 영상 제목
- 답변 전문(answer_full_text)
  - 사용자에게 다시 보여줄 답변 텍스트

- 피드백 텍스트

13. 보안 및 저장 정책
- 모든 주요 기능은 로그인 사용자만 접근
- 업로드 파일은 공개 URL이 아니라 protected storage key로 관리
- 다운로드는 인증 확인 후 제공
- 면접 영상 원본과 raw frame image는 장기 저장하지 않음
- raw transcript, raw vision metrics, hidden score, 세션 중간 상태, 실패 세션 임시 데이터는 세션 종료 후 10분 내 삭제
- 사용자에게 가치가 있는 문서/리포트/세션 결과는 저장

14. 왜 이렇게 저장 구조를 나눴는가
- 사용자는 지난 자소서 피드백과 면접 결과를 다시 보고 싶어한다.
- 하지만 영상 원본과 raw 분석값은 민감하고 무겁다.
- 그래서 “다시 볼 가치가 있는 결과물”은 저장하고, “민감하고 무거운 원본/중간값”은 장기 저장하지 않는 구조로 설계했다.

15. 발표/면접에서 강조할 수 있는 포인트
- 단순 AI 호출이 아니라 취업 준비 전체 흐름을 고려한 서비스형 설계
- 텍스트 중심 면접 평가에 Vision 보조 평가를 결합한 멀티모달 구조
- 영상 원본은 저장하지 않고 답변 전문과 최종 결과만 저장하는 보안/비용 고려
- 사용자 문서, 공고 분석, 자소서 리포트, 면접 세션 이력이 연결되는 실서비스형 데이터 구조

16. 향후 확장 포인트
- AI Agent 기반 면접 오케스트레이션 고도화
- 실시간 면접 모드

17. 최종 정리
- 이 프로젝트는 취업 준비생이 문서와 면접을 반복적으로 개선할 수 있는 서비스다.
- 자소서와 면접 결과는 다시 볼 수 있어야 하므로 저장한다.
- 다만 영상 원본과 raw 내부값은 장기 저장하지 않는다.
- 따라서 DB는 사용자 결과물과 운영 메타를 저장하고, 임시 저장소는 AI 내부 계산값을 짧게 보관하는 구조가 가장 적절하다.
